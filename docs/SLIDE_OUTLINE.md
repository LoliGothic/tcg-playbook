# Duel Matrix 発表スライド骨子

このドキュメントは，最終発表（Week 15）用スライドの骨子です．
各スライドに「見せる内容」と「話すことメモ」を付けているので，そのまま資料化できます．
発表時間の目安は 1 枚あたり 30 秒〜1 分です．

---

## スライド 1: タイトル

- 見せる: プロジェクト名「Duel Matrix」，サブタイトル（デュエマ勝率マトリクスアプリ），チーム名，メンバー名．
- 話す: 何のアプリかを一言で（「対戦記録から相性を数値化するアプリです」）．

## スライド 2: プロジェクト概要

- 見せる: マトリクス表示のイメージ図か表．
- 話す: 対戦記録を登録すると，アーキタイプ同士の勝率が表で見られること．

## スライド 3: テーマ選定理由

- 見せる: 「なぜこのテーマ？」の箇条書き．
- 話す: 身近な題材で，勝率という明確な計算対象があり，OOP・パターンを見せやすいこと．
- 話す: Duel Matrix（今回）と TCG Playbook（将来）の関係も一言で．

## スライド 4: 主な機能

- 見せる: 5 機能（アーキタイプ登録／一覧，対戦記録登録／履歴，勝率マトリクス）．
- 話す: 最小構成に絞った理由（設計の質を見せるため）．

## スライド 5: 全体アーキテクチャ

- 見せる: Controller → Service → Repository の層構成図．
- 話す: 各層の責務と，一方向依存で疎結合にしていること．

## スライド 6: DB 設計

- 見せる: ER 図（`assets/er_diagram.mmd`）．
- 話す: archetypes / matches の 2 テーブル最小構成．外部キー 2 本で自分・相手を表すこと．

## スライド 7: API 設計

- 見せる: 5 つの API 一覧表．
- 話す: REST で画面と分離していること．JSON 例を 1 つだけ見せる．

## スライド 8: デザインパターン

- 見せる: パターン関係図（`assets/patterns_diagram.mmd`）．
- 話す: Factory / Command / Template Method を「なぜ・どこで」使ったか．
  - Factory: 生成の分離
  - Command: 操作の抽象化＋ポリモーフィズム
  - Template Method: マトリクス手順の共通化（継承）

## スライド 9: 担当分担

- 見せる: A / B / C / D の担当表．
- 話す: 非同期開発をどう回したか（Issue → PR → develop）．Git 履歴で貢献が見えること．

## スライド 10: デモ内容

- 見せる: デモの流れ（① アーキタイプ登録 → ② 対戦記録登録 → ③ マトリクス表示）．
- 話す: 実際に `docker compose up` で動かした画面を見せる（動画 or ライブ）．

## スライド 11: 工夫した点

- 見せる: 箇条書き（enum で表記ゆれ防止，Controller を薄く保つ，ゼロ除算回避，Docker で環境統一 など）．
- 話す: OOP 要件（カプセル化・継承・ポリモーフィズム）が実際どこに現れているか．

## スライド 12: 今後の展望

- 見せる: TCG Playbook への発展（PostgreSQL / Go / iOS / 複数 TCG / 調整ノート）．
- 話す: 今回の設計が移植しやすい形になっていること（API・DB をシンプルに保った）．

---

## 発表全体の進め方メモ

- 最初の 1 分で「何のアプリで，何を見せたいか」を言い切る．
- 設計（アーキテクチャ・パターン）に時間を多めに割く（採点観点のため）．
- デモは短く，つまずかないよう事前に手順を固定しておく．
- 担当ごとに話すパートを分けると，全員の貢献が伝わりやすい．
