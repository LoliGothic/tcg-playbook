## 1. プロジェクト概要

Duel Matrix は，デュエル・マスターズの対戦記録を管理し，アーキタイプ同士の
勝率マトリクスを表示する Java アプリケーションです．

ユーザーは，**勝った試合**について，勝ったアーキタイプ・負けたアーキタイプ・（勝者の）先攻後攻・
メモを登録できます（勝者記録モデル）．登録された対戦記録をもとに，「青白ウィリデは創世竜に
何％勝っているか」のような勝率を計算し，表形式で表示します．

本プロジェクトは，将来的に開発予定の TCG 対戦記録アプリ **「TCG Playbook」** の
一部機能を，授業課題用に Java / Spring Boot で実装するプロトタイプです．

### この課題で示したいこと

実用アプリを完全に作ることが目的ではなく，以下を示すことが目的です．

- オブジェクト指向設計（カプセル化・継承・ポリモーフィズム）
- デザインパターンの目的を持った利用
- 疎結合・高凝集・保守しやすい構成
- 4 人で分担しやすい設計
- Git を使ったチーム開発
- Docker / Docker Compose による環境構築
- PDF レポートと発表スライドで説明しやすい構成

### 今回の実装対象

- アーキタイプの登録・一覧
- 対戦記録の登録・履歴表示
- 勝率マトリクスの計算・表示
- 上記を支える REST API と最小限の画面

### 今回の実装対象外

- ログイン・認証・ユーザー管理
- 管理者権限・アーキタイプ申請承認
- CSV 出力の本格実装・外部 API 連携
- iOS アプリ連携・Go 実装・PostgreSQL 本格運用

### なぜ最小構成に絞るのか

授業課題の評価対象は「動く大規模アプリ」ではなく，**設計の質**（OOP・デザインパターン・
コード品質）です．機能を増やすほどコードが複雑になり，設計の意図が読み取りにくくなります．
そこで，勝率マトリクスという 1 つの中心機能に絞り，そこに設計上の工夫を集中させます．

### 将来の展望（今回のスコープ外）

将来 **TCG Playbook** として発展させる際は，以下を検討します（今回は実装しません）．

- 未登録アーキタイプの申請 → 承認・却下フロー
- CSV 出力・PostgreSQL 移行・Go バックエンド移行
- iOS ネイティブアプリ「TCG Playbook」との API 連携
- 複数 TCG 対応・調整ノート機能

そのため今回の API と DB 設計は，**シンプルで移植しやすい形**にしておきます．

## 2. 用語説明

| 用語 | 意味 |
| --- | --- |
| アーキタイプ | デッキの種類．例: 青白ウィリデ，創世竜，黒緑アビス． |
| 対戦記録 | 勝った試合について，勝ったアーキタイプ・負けたアーキタイプ・（勝者の）先攻後攻・メモを保存したもの．勝った試合のみ登録する（勝者記録モデル）． |
| 勝率マトリクス | 行を自分のアーキタイプ，列を相手のアーキタイプとして，対面ごとの勝率を表示する表．A対B の勝率は「A が B に勝った数 ÷（A が B に勝った数 ＋ B が A に勝った数）」． |
| 先攻後攻 | 対戦時に勝者が先攻だったか・後攻だったか・不明だったかを表す情報． |
| Duel Matrix | 今回の授業課題で作成する Java / Spring Boot 版のプロトタイプ名． |
| TCG Playbook | 将来的に開発予定の本番ネイティブアプリ名．TCG 全般の対戦記録・調整ノート・勝率分析を扱う想定． |
| 最小スコープ | 授業提出向けに，実装範囲を小さく絞った構成． |
| full scope | 将来的に本格アプリ化するときの拡張構想． |

## 3. 基本的な利用の流れ

### 3.1 アーキタイプを登録する流れ

1. ユーザーは `/archetypes` 画面を開く．
2. 入力欄に自分が使うデッキの名前（例: 青白ウィリデ）を入力して「追加」を押す．
3. 画面は `POST /api/archetypes` を呼び，登録が成功すると一覧に新しい行が増える．
4. 同じ名前のアーキタイプが既にある場合はエラーが表示され，登録されない．

### 3.2 対戦記録を登録する流れ

1. ユーザーは `/matches` 画面を開く．
2. 勝ったアーキタイプ・負けたアーキタイプをプルダウンから選び，（勝者の）先攻後攻，メモ，
   対戦日時を入力して「記録する」を押す（勝った試合のみ登録する）．
3. 画面は `POST /api/matches` を呼び，登録が成功すると履歴の先頭に追加される．
4. 存在しないアーキタイプを指定した場合，または勝者と敗者が同じ場合はエラーになる．

### 3.3 対戦履歴を見る流れ

1. ユーザーは `/matches` 画面を開く．
2. 画面は `GET /api/matches` を呼び，対戦履歴を新しい順に表示する．
3. 各行には，勝者・敗者のアーキタイプ名，（勝者の）先攻後攻，メモ，対戦日時が並ぶ．

### 3.4 勝率マトリクスを見る流れ

1. ユーザーは `/matrix` 画面を開く．
2. 画面は `GET /api/matrix` を呼び，登録済みの対戦記録から勝率を計算した結果を受け取る．
3. 行＝自分のアーキタイプ，列＝相手のアーキタイプの表として勝率（％）を表示する．
4. 同じアーキタイプ同士のマス（対角線）は「-」で表示する．

## 5. システム構成の概要

```
[ブラウザ] --HTTP--> [Spring Boot アプリ]
                        ├─ ViewController（Thymeleaf 画面）
                        ├─ REST Controller（/api/**）
                        ├─ Service（業務ロジック）
                        ├─ Command / Factory / Template（デザインパターン）
                        └─ Repository（Spring Data JPA） --> [H2 Database]
```

- 画面と API は同一の Spring Boot アプリ内で提供する．
- データは H2 Database（アプリ内）に保存する．
- レイヤは Controller → Service → Repository の順に依存し，逆流させない．

## 6. 課題要件（OOP）との対応

各要件を，どのクラス・どの設計・どの図で満たすかを対応づけます．
詳細なコードレベルの説明は [CLASS_DESIGN.md](CLASS_DESIGN.md) を参照してください．

| 要件 | 実現方法 | 主な該当クラス | 参照図 |
| --- | --- | --- | --- |
| カプセル化 | Entity のフィールドを `private` にし，getter や Factory・Service 経由でのみ操作する | `Archetype`, `MatchRecord` | class_diagram.mmd |
| 継承 | `AbstractMatrixBuilder` を `StandardMatrixBuilder` が継承する | `AbstractMatrixBuilder`, `StandardMatrixBuilder` | class_diagram / patterns_diagram |
| ポリモーフィズム | `Command<T>` インターフェースを通じて異なる操作を `execute()` で同一に扱う | `CreateArchetypeCommand` ほか | patterns_diagram.mmd |
| デザインパターン（2〜3種） | Factory / Command / Template Method を実際の設計課題に対して採用する | pattern.* パッケージ | patterns_diagram.mmd |
| パターンの正当化 | 各パターンの採用理由を CLASS_DESIGN と REPORT_OUTLINE に記述 | — | — |
| 疎結合 | Controller は Service / Command を呼び，Service は Repository / Factory / Template を使う（層をまたいだ直接依存を避ける） | 全層 | class_diagram.mmd |
| 高凝集 | Service を機能ごとに分割（Archetype / Match / Matrix） | `*Service` | class_diagram.mmd |
| 変更に強い設計 | 勝率計算・オブジェクト生成を独立クラスに分け，変更の影響範囲を局所化する | Factory / Template | patterns_diagram.mmd |

### なぜ API サーバでも「ソフトウェアシステム」として成立するのか

REST API サーバは，HTTP リクエストを受け取り，業務ロジックを実行し，データを永続化して
応答を返す，独立したソフトウェアシステムです．画面（Thymeleaf）も同居しているため，
入力から表示までの一連の流れが 1 つのアプリケーションとして完結します．
GUI アプリと同様に，クラス設計・カプセル化・パターン適用の対象になります．

## 7. 4 人の担当分担

| 担当 | 範囲 | 詳細文書 |
| --- | --- | --- |
| A | DB・Entity・Repository・Docker 環境構築 | [TASK_A_REPOSITORY.md](TASK_A_REPOSITORY.md) |
| B | アーキタイプ登録・一覧機能（Factory / Command） | [TASK_B_ARCHETYPE.md](TASK_B_ARCHETYPE.md) |
| C | 対戦記録登録・履歴機能（Factory / Command） | [TASK_C_MATCH.md](TASK_C_MATCH.md) |
| D | 勝率マトリクス計算・画面表示（Command / Template Method） | [TASK_D_MATRIX_UI.md](TASK_D_MATRIX_UI.md) |

A が基盤（DB・Entity・Repository・コマンド基盤・Docker）を先に固め，`develop` にマージしてから，
B・C・D が並行して機能開発を進めます．詳細は [DEVELOPMENT_GUIDE.md](DEVELOPMENT_GUIDE.md) 参照．

### 担当間の依存関係（重要な設計方針）

**B・C・D は互いに依存せず，A だけに依存します．** これにより 3 人が完全に並行して開発できます．

| 依存元 | 依存先 | 内容 |
| --- | --- | --- |
| B | A のみ | Entity・`ArchetypeRepository`・コマンド基盤 |
| C | A のみ | Entity・`MatchRepository`・`ArchetypeRepository`（存在確認）・コマンド基盤 |
| D | A のみ | Entity・両 Repository（マトリクス計算）・コマンド基盤 |

- C の対戦記録登録時のアーキタイプ存在確認は，B の Service ではなく **A の `ArchetypeRepository`** を使う．
- D のマトリクス計算は，B・C の API ではなく **A の Repository** から直接データを取得する．
- D の画面が呼ぶ API（`GET /api/archetypes` / `GET /api/matches`）は，コードではなく
  [API_SPEC.md](API_SPEC.md) で先に固定した**契約**への依存なので，並行開発の妨げにならない．
